# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Lens** — an MCP (Model Context Protocol) server, implemented in `server.py` (registered with FastMCP under the name `"Lens"`), that exposes two tools:

- `web_search(query, max_results=3, engines="all", output_format="text")` — searches across multiple engines (DuckDuckGo, Naver news, Naver web documents, Naver encyclopedia/dictionary) concurrently and returns titles/URLs/content grouped by engine. The top results per call get their full article text fetched and cleaned (via `trafilatura`), not just a short snippet — the tool's own output already looks like a Tavily-style `{title, url, content}` result, not a bare snippet list. `output_format="json"` returns a structured `{"query", "engines", "results": [...]}` payload instead of the human-readable text block.
- `search_stats()` — reports how many times each engine (and each individual Naver key) has been called today, for quota monitoring.

The server supports two transports, chosen via `.env`'s `MCP_TRANSPORT`: `stdio` (default — launched by an MCP client via command+args, no network port) or `sse`/`streamable-http` (listens on `MCP_HOST:MCP_PORT`, registered by URL, shareable across clients). See `README.md` for setup/registration instructions (including the directory-tree overview and the exact `mcp.run()` startup sequence/stdio-vs-network process-lifecycle breakdown) and `PLAN.md` for the full design writeup and verification history. Repo: [github.com/ChoiDongHee/ktds_lens](https://github.com/ChoiDongHee/ktds_lens).

`test/` holds a real end-to-end MCP protocol test (`test/test_mcp_client.py`, the only thing here that actually speaks the MCP protocol), a FastAPI demo chat app (`test/webapp/app.py`) used to manually compare this server's search quality against Tavily's own MCP server (side-by-side "compare" mode runs both concurrently and shows two answers), grounding a vLLM-hosted Qwen model's answers on either backend, and a benchmark/quality report script for `bm25_search` (`test/test_bm25_benchmark.py`, results in `README.md`). None of these are part of the MCP server itself — keep test-only logic there, not in `server.py`. `test/webapp/app.py` imports `server` directly and calls `server.router.search()` as a plain function (no protocol round-trip) — its `search_mcp()` wraps that call plus its own BM25 relevance re-filter (`_filter_relevant_links`) in a `try/except` that falls back to `engines="duckduckgo"` (skipping the relevance filter on the fallback too) and logs the failure, rather than letting an unhandled exception surface to the browser as a chat request failure.

`bm25_search/` is a standalone module (`fetcher.py` → `tokenizer.py` → `bm25_ranker.py` → `ranker.py` → `pipeline.py`, run via `python -m bm25_search.pipeline "<query>"`) that fetches Naver News candidates and reranks them by BM25 keyword-similarity combined with a recency-decay factor (`bm25_score * exp(-decay_lambda * hours_since_published)`); `tokenizer.py` uses a single module-level `Kiwi` instance since loading the model is the expensive part. Its `tokenizer`/`bm25_ranker` pieces are also imported directly by `server.py` (`_filter_by_relevance`, see below) and by `test/webapp/app.py` (`_filter_relevant_links`) for relevance filtering — the standalone `pipeline.py` (candidate fetch + recency-aware top-K) and the relevance filter used by both consumers are two different uses of the same underlying BM25 building blocks. See `README.md`'s bm25_search section for the measured performance/quality report.

`tokenizer.py`'s `tokenize()`/`tokenize_batch()` handle two things Kiwi alone doesn't: (1) **URLs** are pulled out via regex *before* Kiwi sees the text and decomposed into domain/path fragments (`_url_tokens`, e.g. `"https://www.samsung.com/sec/"` → `["samsung", "sec"]`) — Kiwi tags a whole URL as a single non-content token outside `_KEEP_TAGS`, so without this a query that's just a URL tokenizes to `[]`, which silently disables `_filter_by_relevance` entirely (its `if not tokenized_query: return` guard) rather than erroring; found via a deliberate review of "what happens when a URL/specific site is the query" and verified fixed with both synthetic cases and a real `naver_encyc` call (its results, previously unfiltered for that query, are now correctly dropped). (2) **all tokens are lowercased** — Kiwi preserves original case on foreign-word (`SL`) tokens (e.g. `"Samsung"` vs `"samsung"`), and BM25 matches by exact string equality, so without normalizing case the URL-derived (already-lowercased) query tokens would never match a differently-cased form of the same word in a document.

## Environment

- Python 3.12, managed via a local venv at `.venv/`.
- Dependencies are listed in `requirements.txt`: `mcp`, `ddgs`, `httpx`, `python-dotenv`, `anyio`, `trafilatura`, `kiwipiepy`, `rank_bm25` (the last two are `bm25_search`'s, imported directly by `server.py` for relevance filtering — not optional).
- Configuration lives in `.env` (see `.env.example` for the full list with samples and explanations) — nothing is hardcoded. Every tunable (Naver keys/quota, which engines run by default, content-fetch behavior, excluded keywords, transport, logging, the test webapp's vLLM/Tavily settings) is an environment variable with a sane default.
- A previously-tried `GoogleEngine` (headless-Chromium Google scraping via `noapi-google-search-mcp`) was deliberately removed — too heavy (~500MB of Playwright/OpenCV/ONNX/faster-whisper deps), too slow (~20s/call), fragile to Google's bot detection, and it broke under async/SSE transport (`asyncio.run()` inside an already-running event loop). The removed code is preserved in `backup/google_engine_removed.py` with restore notes.

## Running

```
.venv/Scripts/python.exe server.py
```

With the default `stdio` transport this blocks waiting for a client connection — no visible stdout output on its own (stdout is reserved for the MCP protocol). With `MCP_TRANSPORT=sse` or `streamable-http` it instead starts a Uvicorn server on `MCP_HOST:MCP_PORT` and logs startup to `LOG_FILE`/stderr.

For quick manual testing without going through the protocol, import `server` directly in a Python REPL/script and call `server.router.search(query, max_results, engines=..., output_format=...)` / `server.router.stats_report()`.

For a real end-to-end protocol test (launches the actual server process, connects over SSE, calls tools like a genuine MCP client), run `test/test_mcp_client.py`. For an interactive comparison of this server's search quality vs. Tavily's, run `test/webapp/app.py` and open the printed local URL.

## Architecture

- **Engines** (`SearchEngine` subclasses), each normalizing to a common `{"title", "url", "snippet"}` schema:
  - `DuckDuckGoEngine` — free, keyless, no quota.
  - `NaverEngine` — hits one Naver Open API search vertical (`_endpoint`) using N Client ID/Secret pairs loaded from `.env`, rotating to the next key once one nears `NAVER_LIMIT_THRESHOLD`. Up to three instances share the same credentials/quota (same underlying Naver app): `"naver"` (news, `NAVER_SEARCH_ENDPOINT`), `"naver_web"` (general web documents, `NAVER_SEARCH_ENDPOINT_2`), `"naver_encyc"` (encyclopedia/dictionary, `NAVER_SEARCH_ENDPOINT_3` — much better than news/web-doc search for definitional "what is X" queries). `name`/`endpoint` are constructor params specifically so more instances can reuse the class against a different vertical without duplicating the quota-rotation logic; set any `NAVER_SEARCH_ENDPOINT_*` to empty to disable that instance.
  - `SEARCH_ENGINE_MODE` in `.env` is a preset for `DEFAULT_ENGINES`: `"hybrid"` (everything, including DuckDuckGo), `"naver"` (default — Naver verticals only), `"duckduckgo"` (DuckDuckGo only). Setting `DEFAULT_ENGINES` explicitly always overrides the preset. **DuckDuckGo (via `ddgs`) is excluded from the default set on purpose** — measured 4.9s average / 8.9s worst case (it internally tries several scraping backends), vs. ~0.25-1.5s for Naver-only configs; still reachable via `engines="duckduckgo"`. See README.md's "검색 속도 최적화" for the full measured comparison (including the `FETCH_CONTENT` on/off tradeoff — enabling it adds ~0.5-0.8s regardless of engine set, since it's a real second network hop to fetch each article's own page).
- **Keyword filtering** (`EXCLUDE_KEYWORDS` in `.env`, default covers common sports terms): each engine over-fetches (`EXCLUDE_OVERFETCH_COUNT` raw results instead of just `max_results`) and drops any result whose title+snippet matches an excluded keyword before returning `max_results`. If every single raw result gets excluded, the engine raises `PolicyRestricted` (distinct from "the API found nothing") which the router treats like any other engine failure — falls back to another engine, doesn't crash. Note this is a literal substring match: querying about the excluded topic itself (e.g. "프로야구 오늘 경기 결과") gets most results filtered out too — verified as the intended, working behavior, not a bug.
- **BM25 relevance filtering** (`RELEVANCE_FILTER`/`RELEVANCE_MIN_SCORE_RATIO` in `.env`, default on): before content enrichment, `SearchRouter.search()` calls `_filter_by_relevance()`, which merges every engine's combined results, tokenizes each result's title+description (the text engines already returned — no extra fetch) and the query via `bm25_search.tokenizer`, scores them with `bm25_search.bm25_ranker` (same BM25 model the standalone `bm25_search` pipeline uses), and drops anything scoring below `best_score * RELEVANCE_MIN_SCORE_RATIO`. This is what stops a vertical tuned for a different kind of query (typically `naver_encyc`) from occasionally surfacing something that only shares one incidental word with the query (e.g. a "주가"/stock-price query pulling in a "영구 제모 효과"/permanent-hair-removal dictionary entry) — verified with a synthetic case reproducing exactly that. Runs *before* content enrichment specifically so irrelevant results don't waste a fetch slot.
- **PDF exclusion** (`_filter_out_pdfs`, no on/off switch): runs right after engine results are gathered, before both the relevance filter and content enrichment — any result whose URL matches `_looks_like_pdf` (a `.pdf` extension check) is dropped from `results_by_engine` entirely. Not just a content-enrichment skip: found live that Naver's own search snippet for a PDF result is just as unreadable as our own `trafilatura` extraction of one (a raw, out-of-context text fragment either way), so there's nothing worth showing for a PDF hit regardless of source.
- **Content enrichment** (`FETCH_CONTENT*` in `.env`): after PDF exclusion and relevance filtering, `SearchRouter.search()` flattens the surviving combined results and calls `_enrich_with_content()`, which fetches (concurrently, via a thread pool) and cleans (`trafilatura`, with `include_comments=False`/`favor_precision=True` plus a small regex pass for lingering copyright/comment-label boilerplate) full article text for up to `FETCH_CONTENT_TOP_N` results *across all engines combined* — not per engine. Candidates that yield too little real content (`FETCH_CONTENT_MIN_USEFUL_CHARS`, typically JS-rendered pages) are skipped in favor of the next candidate rather than wasting a content slot; `_looks_like_pdf` is checked again here too as defense-in-depth (harmless redundancy since `_filter_out_pdfs` already removed them upstream). `trafilatura.fetch_url()` is called with a custom `Config` (`_TRAFILATURA_CONFIG`, built via `trafilatura.settings.use_config()`) overriding `DOWNLOAD_TIMEOUT` to `FETCH_CONTENT_TIMEOUT_SECONDS` (default 5s) instead of trafilatura's own 30s default — found live while testing URL-as-query cases: since `_enrich_with_content` submits every candidate to the same thread pool and blocks until they've all resolved, one candidate happening to be a large PDF/binary (e.g. a security-guide PDF a Naver web-document result linked to) took 37.9s to fetch and single-handedly stalled the whole `web_search` call to ~38s; capped to ~5s worst-case after the fix (re-measured: 38.475s → 0.548s for the same query once the slow fetch was no longer on the critical path). `SEARCH_ENGINE_TIMEOUT_SECONDS` (per-engine API call timeout) was tightened alongside this from 15s to 8s.
- **`SearchRouter`**: `_run_batch()` submits every requested engine's `search()` to a `ThreadPoolExecutor` and runs them **concurrently** (they're independent services — sequential execution would just add their latencies together; measured ~3.5s for 4 engines concurrently vs. what would be ~8-10s sequential), each bounded by `SEARCH_ENGINE_TIMEOUT_SECONDS` (a slow engine is logged and skipped, not left to block the whole request — note the executor is *not* used as a context manager, since `__exit__`'s `shutdown(wait=True)` would block on stragglers we already gave up on; `shutdown(wait=False)` lets them finish in the background instead). If, after that first round, no engine produced any results, one further concurrent round runs every other *registered* engine not yet tried as a fallback (`EngineUnavailable`/`QuotaExceeded`/`PolicyRestricted` are the expected reasons to fall back; this only fires while no engine has produced results yet — a missing/exhausted Naver key shouldn't also trigger fallback once another engine already returned a perfectly good answer). `engines="all"` expands to `DEFAULT_ENGINES`, not literally every registered engine.
- **`CallStats`**: persists per-key daily call counts to `search_stats.json`, keyed by engine name (`duckduckgo`) or per-Naver-key name (`naver_1`, `naver_2`, ...) — note `naver` and `naver_web` share the same `naver_N` keys since they're the same Naver app. Guarded by a `threading.Lock` since concurrent SSE/HTTP requests run their engine calls in separate worker threads. `reset_stale_entries()` actively zeroes out counters whose stored date has passed, called both at startup and on every hit-count tick (see below) so quotas free up once a day even with the server idle at midnight.
- **Concurrency**: FastMCP calls sync tool functions directly on its event loop thread — it does **not** offload them to a worker thread the way it does for resources. Since `router.search()` does blocking network I/O, `web_search` is defined `async` and offloads the actual work via `anyio.to_thread.run_sync(router.search, ...)`, so one slow/concurrent request doesn't stall every other client under `sse`/`streamable-http` transport. (Verified: reproduced the stall/crash before the fix, re-measured 8 concurrent calls afterward — ~4s wall time instead of the ~17s a serial fallback would take.)
- **Exception hardening**: both `@mcp.tool()` functions wrap their entire body in `try/except Exception`, always returning a Korean error string rather than letting anything propagate as an MCP tool error. Content enrichment and JSON serialization each have their own inner `try/except` so a failure there degrades to snippet-only/plain-text output instead of failing the whole call.
- **Logging**: `stdout` is reserved for the MCP stdio protocol, so all logs go to `LOG_FILE` (default `search.log`) and `stderr` only — never print to stdout from this codebase. Every `web_search` call gets an 8-char `request_id` tying together all its log lines (engine attempts, Naver key/quota checks, keyword exclusions, BM25 relevance-filter drops, content-fetch outcomes, fallback decisions, timings, the final rendered per-engine result). A one-line startup summary (`"Lens 서버 설정: ..."`, logged right after `router` is constructed) records the resolved engine list, what `engines="all"` expands to, and whether relevance filtering/content enrichment are on — so a `search.log` reader doesn't need `.env` open to know what a given request *should* have done. A separate daemon thread (`_hit_count_loop`) ticks every `HIT_COUNT_INTERVAL_SECONDS` and writes current usage to both `HIT_COUNT_LOG_FILE` (dedicated) and the main log; its body is wrapped in its own `try/except` so one bad tick (e.g. a `CallStats` disk I/O failure) logs and retries next interval instead of silently killing the thread forever. `CallStats._load()`/`_save()` similarly catch `OSError` around the `search_stats.json` read/write so a disk hiccup degrades to an unpersisted quota count rather than failing the engine call that triggered it. README.md's "로그 읽는 법" section documents the specific log strings to grep for when diagnosing an issue.
- **Windows encoding**: `sys.stdout`/`sys.stderr` are force-reconfigured to UTF-8 at startup, because Windows defaults to the system codepage (e.g. `cp949` on Korean Windows) which cannot represent arbitrary search-result Unicode and would otherwise crash the stdio protocol (or any script printing results) mid-response. Any new script that prints raw tool output needs the same fix (see `test/test_mcp_client.py`, `test/webapp/app.py`).
