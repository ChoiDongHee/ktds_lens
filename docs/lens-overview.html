Lens

여러 검색엔진에서 들어오는 결과를 하나의 초점으로 모으는 통합 웹 검색 MCP 서버. 엔진들은 전부 동시에(스레드) 실행되고, 하나가 흐릿하면(실패/한도초과) 자동으로 다른 엔진에 초점을 맞춥니다.

개요

Lens는 DuckDuckGo와 네이버(뉴스/일반 웹문서/백과사전)를 하나의 MCP 툴로 묶은 검색 서버입니다. 여러 렌즈(엔진)를 동시에 열어 놓고, 가장 먼저 초점이 맞는 결과를 취하되 필요하면 다른 렌즈로 자동 전환합니다.

동시 실행
요청된 모든 엔진을 스레드풀로 한 번에 실행. 순차 실행 대비 최대 4배 이상 빠름.
자동 폴백
엔진 실패·한도초과·불용어 전체 매칭 시, 결과가 없을 때만 다른 엔진으로 자동 전환.
BM25 관련성 필터 (내장)
엔진 합산 결과를 검색어와 BM25로 비교해, 우연히 한 단어만 겹치는 무관한 결과를 본문을 가져오기도 전에 제외. MCP 툴 자체에 내장되어 어떤 클라이언트로 붙어도 동일하게 적용됨.
본문 보강
관련성 필터를 통과한 상위 결과는 스니펫이 아니라 trafilatura로 추출한 실제 기사 본문을 반환.
불용어 필터
지정 키워드(기본: 스포츠 용어)가 섞인 결과를 자동 제외, 전부 걸리면 다른 엔진으로 폴백.
키 순환
네이버 키를 여러 개 등록하면 하루 한도 근접 시 자동으로 다음 키로 순환.
BM25 서브시스템
bm25_search/는 뉴스 후보군을 키워드 유사도 + 최신성으로 재정렬하는 독립 랭킹 파이프라인이자, 위 관련성 필터의 BM25 코어 자체이기도 함.
제공 툴 web_search(query, max_results=3, engines="all", output_format="text") — 검색 실행
search_stats() — 오늘 엔진·키별 호출 현황 및 남은 한도 조회

검색 엔진

모든 엔진은 {"title", "url", "snippet"} 공통 스키마로 정규화되어 SearchRouter가 출처와 무관하게 동일한 방식으로 합치고 표시합니다.

엔진 이름출처키 필요비고
duckduckgoddgs 라이브러리불필요내부적으로 여러 백엔드를 순회해 느리고 편차가 커서 기본 목록에서는 제외됨
naver네이버 오픈API news.json필요뉴스/시사성 질의에 강함, originallink(원본 기사 URL) 사용
naver_web네이버 오픈API webkr.json필요일반 웹문서 — 실시간 시세 페이지, 공식 IR 페이지 등 뉴스가 놓치는 것 보완
naver_encyc네이버 오픈API encyc.json필요백과사전/용어사전 — "OO가 뭐야?" 류 정의성 질문에 강함

세 네이버 엔진은 같은 앱(Client ID/Secret)을 쓰기 때문에 일일 호출 한도를 공유합니다. 키를 여러 개 등록(NAVER_CLIENT_IDS에 콤마로 나열)하면, 한 키가 한도의 NAVER_LIMIT_THRESHOLD(기본 80%)에 도달했을 때 다음 키로 자동 순환합니다.

폴백 로직

엔진이 다음 세 가지 이유로 실패하면 결과가 아직 하나도 없을 때만 나머지 등록된 엔진으로 한 차례 더 동시 폴백을 시도합니다.

  • EngineUnavailable — 네이버 키가 설정되어 있지 않음
  • QuotaExceeded — 등록된 모든 네이버 키가 한도 근접
  • PolicyRestricted — 원본 결과 전체가 불용어에 걸려 제외됨
중요한 설계 포인트 이미 다른 엔진에서 결과를 받았다면 폴백은 일부러 생략됩니다. 예를 들어 DuckDuckGo가 이미 결과를 반환했는데 네이버 키가 없다고 해서 또 다른 무거운 엔진을 추가로 돌리지 않습니다 — 실제로 이 조건이 빠져 있었을 때 응답시간이 8.9초까지 늘어지는 버그를 발견하고 고친 이력이 있습니다.

BM25 관련성 필터 & 본문 보강 & 불용어 필터

모든 엔진의 결과를 합친 뒤, 본문을 가져오기 전에 RELEVANCE_FILTER(기본 켜짐)가 먼저 실행됩니다: 각 결과의 제목+설명(엔진이 이미 돌려준 텍스트, 추가 요청 없음)을 bm25_search와 동일한 BM25 모델(tokenizer.py/bm25_ranker.py를 그대로 import)로 검색어와 비교하고, 이번 배치 최고 점수 대비 RELEVANCE_MIN_SCORE_RATIO(기본 0.15) 미만인 결과는 제외합니다. 여러 검색 종류를 합치면 성격이 다른 벡터(특히 naver_encyc)가 검색어와 한 단어만 우연히 겹치는 결과(예: 주가 질문에 "영구 제모 효과" 사전 항목)를 섞는 경우가 있는데, 이 필터가 본문 보강보다 먼저 실행되어 애초에 관련 없는 링크에 본문 요청을 낭비하지 않습니다.

필터를 통과한 결과 중 상위 FETCH_CONTENT_TOP_N개(기본 3개)는 스니펫 대신 실제 페이지 본문을 trafilatura로 추출해 채워 넣습니다. include_comments=False와 favor_precision=True 옵션 + 정규식 후처리로 댓글·저작권 문구 같은 비본문 텍스트를 제거합니다. 추출된 글자 수가 FETCH_CONTENT_MIN_USEFUL_CHARS(기본 200자) 미만이면(대개 JS 렌더링 페이지) 그 후보를 건너뛰고 다음 후보로 넘어갑니다.

불용어 필터(BM25 관련성 필터보다 앞선 단계, 각 엔진 내부에서 실행)는 각 엔진이 max_results보다 넉넉히(기본 10건) 가져온 뒤, 제목+스니펫에 EXCLUDE_KEYWORDS(기본: 야구·축구·올림픽 등 스포츠 용어)가 있으면 제외하고 남은 것 중 상위 개수만 반환합니다.

쓰레드 구조 가이드 동시성 설계 이해하기

Lens 안에서 스레드가 정확히 어디서, 왜 쓰이는지를 하나씩 짚어봅니다. 코드를 처음 보는 분도 따라올 수 있도록 "무엇이 문제였고, 어떻게 풀었는지" 순서로 설명합니다.

왜 스레드가 필요한가

검색 엔진 하나하나는 결국 외부 서버에 HTTP 요청을 보내고 응답을 기다리는 일입니다. 이런 작업(I/O 대기)은 CPU를 쓰는 게 아니라 그냥 네트워크 응답이 올 때까지 멈춰 있는 시간이 대부분입니다. 그런데 만약 엔진 4개를 순서대로(하나 끝나면 다음 시작) 호출한다면, 각자 0.2~2초씩 걸리는 응답을 전부 더한 시간만큼 기다려야 합니다.

스레드를 쓰면 "기다리는 동안 다른 일도 같이 기다리게" 만들 수 있습니다. 4개 엔진을 동시에 쏘아 놓고, 가장 오래 걸리는 하나만큼만 기다리면 되는 것이죠. 실제로 이 프로젝트에서 측정한 수치가 이 효과를 잘 보여줍니다.

상황순차 실행이었다면실제(스레드 동시 실행)
동시 요청 8개 (SSE 트랜스포트 테스트)~17.4초4.07초
기본 엔진 4개(구 버전, 덕덕고 포함)~8~10초 추정3.49초

엔진 동시 실행과 타임아웃 — SearchRouter._run_batch

실제로 엔진들을 동시에 돌리는 코드는 이렇게 동작합니다:

1

ThreadPoolExecutor에 한꺼번에 제출

요청된 엔진 개수만큼 워커를 가진 풀을 만들고, 각 엔진의 search() 호출을 전부 pool.submit()으로 동시에 넘깁니다. 이 시점에서 엔진들은 이미 병렬로 실행되기 시작합니다.

2

결과를 기다리되, 한없이 기다리지 않는다

각 결과는 future.result(timeout=SEARCH_ENGINE_TIMEOUT_SECONDS)로 받습니다. 기본값은 15초 — 이 시간 안에 응답이 없는 엔진은 "시간 초과"로 기록하고 건너뜁니다. 다른 엔진들은 영향받지 않습니다.

3

풀을 닫을 때 wait=False를 쓰는 이유

여기가 실수하기 쉬운 부분입니다. with ThreadPoolExecutor() as pool: 처럼 with문을 쓰면, 블록을 빠져나갈 때 shutdown(wait=True)가 자동으로 호출되어 아직 안 끝난 스레드를 전부 기다립니다 — 방금 타임아웃으로 포기한 그 느린 엔진까지 포함해서요. 그러면 타임아웃을 걸어둔 의미가 없어집니다. 그래서 Lens는 with문 대신 pool.shutdown(wait=False)를 명시적으로 호출합니다. 느린 스레드는 백그라운드에서 계속 실행되다 스스로(자신의 HTTP 타임아웃으로) 끝나고, 우리 응답은 그걸 기다리지 않고 먼저 나갑니다.

4

결과가 없을 때만 한 번 더 — 폴백 라운드

1차 실행이 끝났는데 결과가 하나도 없으면, 아직 시도 안 한 나머지 등록 엔진들을 또 한 번 동시에 실행합니다. 반복적으로 큐를 도는 구조가 아니라 "필요하면 딱 한 라운드 더"이기 때문에 구조가 단순하고 예측 가능합니다.

5

공유 상태는 잠금(Lock)으로 보호

엔진들이 동시에 호출 횟수를 기록(CallStats.record())할 수 있기 때문에, 이 카운터는 threading.Lock으로 감싸져 있습니다. 잠금이 없으면 두 스레드가 동시에 "지금 카운트는 5야" 읽고 "6이 됐어" 저장하면서 실제로는 한 번만 반영되는(lost update) 문제가 생길 수 있습니다. 본문 보강 단계(_enrich_with_content)에서 여러 페이지를 동시에 가져올 때도 같은 이유로 스레드풀을 씁니다 — 다만 그쪽은 서로 다른 URL을 읽기만 하므로 별도 잠금은 필요 없습니다.

왜 그냥 하나씩 처리하면 안 되나요? 네트워크 대기 시간은 CPU가 쉬는 시간입니다. 파이썬의 GIL(Global Interpreter Lock)이 CPU 연산은 한 번에 한 스레드만 돌리게 막지만, I/O 대기 중에는 GIL이 풀리기 때문에 다른 스레드가 그 사이 자기 네트워크 요청을 보낼 수 있습니다. 그래서 이런 "여러 곳에 요청 보내고 기다리는" 작업은 스레드만으로도 충분히 병렬 효과를 볼 수 있습니다.

FastMCP와 이벤트 루프 사이의 다리 — anyio.to_thread.run_sync

MCP 서버 프레임워크(FastMCP)는 동기(sync) 함수로 만든 툴을 이벤트 루프 스레드에서 직접 호출합니다. 리소스와 달리 별도 워커 스레드로 자동으로 옮겨주지 않습니다. 이게 왜 문제가 될까요?

  • stdio 트랜스포트에서는 한 번에 요청이 하나뿐이라 문제가 되지 않습니다.
  • 하지만 sse/streamable-http 트랜스포트는 여러 클라이언트가 동시에 접속할 수 있는데, 검색 하나가 이벤트 루프를 블로킹하면 그 사이 다른 모든 클라이언트의 요청이 멈춰버립니다.

그래서 web_search 툴 함수는 async def로 선언하고, 실제 작업은 await anyio.to_thread.run_sync(router.search, ...)로 별도 워커 스레드에 위임합니다. 이렇게 하면 이벤트 루프는 즉시 다음 클라이언트를 처리하러 돌아갈 수 있습니다. 이 수정 전후로 실제 재현·측정한 결과가 위 표의 "동시 요청 8개" 행입니다.

MCP 클라이언트 등록

기본값인 stdio 방식은 URL이 없습니다. 클라이언트가 명령어로 직접 프로세스를 띄우고 표준입출력으로 통신합니다.

Claude Code

claude mcp add --transport stdio web-search -- "<venv>\Scripts\python.exe" "<project>\server.py"

네트워크로 띄우려면(MCP_TRANSPORT=sse 또는 streamable-http 설정 후):

claude mcp add --transport http web-search http://127.0.0.1:8000/mcp

Claude Desktop

{
  "mcpServers": {
    "web-search": {
      "command": "<venv>\\Scripts\\python.exe",
      "args": ["<project>\\server.py"]
    }
  }
}

sse/streamable-http로 띄운 원격 서버는 Claude Desktop이 URL을 직접 지원하지 않아 mcp-remote 같은 브릿지가 필요합니다.

BM25 유사도 검색 bm25_search/

pipeline.py는 네이버 뉴스 후보군을 가져와 BM25 키워드 유사도 × 최신성 감쇠로 재정렬해 상위 4건만 남기는 독립 CLI/스크립트입니다. 그 안의 tokenizer.py/bm25_ranker.py 두 모듈은 별도 공용 부품이기도 해서, server.py의 web_search 툴 자체와 test/webapp 데모 챗 둘 다 그대로 가져다 씁니다 — 검색엔진 여러 개를 합치면 가끔 완전히 엉뚱한 결과(예: 주가 질문에 "영구 제모 효과" 같은 백과사전 항목)가 섞이는데, 각 결과의 제목+설명(엔진이 이미 돌려준 텍스트, 추가 요청 없음)을 같은 BM25 모델로 검색어와 비교해 본문 보강 이전에 걸러냅니다. 이 필터는 데모 페이지에만 있는 게 아니라 MCP 서버 자체에 내장되어 있어, Claude Desktop 등 어떤 MCP 클라이언트로 붙어도 동일하게 적용됩니다 (RELEVANCE_FILTER/RELEVANCE_MIN_SCORE_RATIO, 설정 가이드 참고).

네이버 뉴스 실 API로 측정 (Kiwi 워커풀 워밍업 후). 재현: test/test_bm25_benchmark.py.

후보 30~100건 전 구간에서 100ms 목표를 여유 있게 만족하며, 후보 개수에 거의 선형으로 비례합니다.

2) 속도 — 대상 텍스트 범위에 따른 차이 (후보 30건, 5회 중앙값)

대상 텍스트tokenize_batchbm25+rerank합계
title + 설명(스니펫)만11.6ms1.1ms12.7ms

전체 본문을 포함하면 텍스트량이 늘어 랭킹 자체는 150ms 안팎이 되지만, 이 모드에서는 기사 원문을 가져오는 네트워크 fetch(30건 기준 2~4초, 스레드풀 동시 실행)가 전체 시간을 지배하므로 실질적 영향은 적습니다.

3) 품질 — 정렬 방식별 질의어 일치도

정렬 방식평균 질의어 일치 수 (상위 4건)
네이버 기본 순서 (그대로)1.75
BM25만 적용2.00
BM25 + 최신성 감쇠2.00

BM25 재정렬이 네이버 기본 순서보다 질의어와 실제로 더 많이 겹치는 문서를 상위로 올리는 것을 확인했습니다. 이 실행에서는 모든 후보가 같은 날짜 뉴스라 최신성 감쇠가 BM25 전용 순서를 바꾸지는 않았는데, 오래된 기사와 최신 기사가 섞인 질의에서는 감쇠가 실제로 순위를 재조정합니다(decay_lambda로 감쇠 속도 조절 가능).

워밍업 비용 참고 Kiwi의 배치 토큰화는 프로세스 내 최초 1회 호출 시 내부 워커풀을 초기화하며 약 1~1.5초의 워밍업 비용이 듭니다. 계속 떠있는 서버 프로세스에서는 이 비용이 최초 1회만 발생하므로 위 수치가 대부분의 실제 호출에 적용됩니다.
